# AGENTS.md

## 使用场景

本项目主要由产品经理在本地使用，用于写产品方案、整理需求、实现可交互原型和验证体验。不默认面向生产发布，除非用户明确说明。

## 核心原则

1. 先理解目标  
   优先确认用户要验证什么：业务目标、用户路径、核心功能、成功标准。若不确定性会影响方向、安全、数据或用户预期，先澄清；普通细节可合理假设并说明。

2. 简洁直接  
   默认用中文，先给结论、方案或结果，再补必要解释。少复述背景，少空话。

3. 原型优先  
   优先做能看懂、能点击、能评审、方便迭代的实现。不要为了工程完美引入复杂架构、重依赖、后端服务或过度抽象。

4. 暴露问题  
   不假装完成。若功能不可用、数据缺失、依赖失败或无法验证，直接说明原因、影响和下一步。

5. 不擅自扩大范围  
   不主动大重构，不改无关文件，不私自创建 branch / worktree。需要扩大范围时先说明并等确认。

## 开发工作流

1. 先判断任务级别  
   - 小改动：文案、样式、布局、交互细节、配置修正，可直接执行。  
   - 中等改动：少量多文件、低风险逻辑调整，先简短列计划再执行。  
   - 大改动：涉及多页面、多流程、目录结构、核心状态、依赖安装、外部接口或数据结构，先规划再执行。

2. 动手前尽量拆成四段  
   `files`：影响哪些文件；`action`：做什么；`verify`：如何验证；`done`：完成后应看到什么。简单任务可不输出完整计划，复杂任务先给简短计划。

3. 高风险操作先确认  
   删除文件、批量覆盖、安装依赖、联网操作、迁移数据、改系统配置、运行不可逆命令、创建 branch / worktree 前，必须先说明风险并等确认。

4. 改完必须验证  
   优先本地运行、构建、类型检查或手动交互验证。若无法验证，说明原因、剩余风险和建议验证方式。

## Skill / 工具使用

1. 以当前会话实际暴露的 skill、tool、agent 为准。本地存在但未暴露的能力，只能作为参考，不能假装已调用。

2. `using-superpowers` 若可用，优先遵循，用于任务拆解、执行纪律和自检。

3. `planning-with-files` 若可用，且任务复杂或需多轮跨文件修改，使用 `task_plan.md`、`progress.md`、`findings.md` 维护计划、进度和发现；不可用时可手工简短记录。

4. `OpenSpec` 只作为需求、变更、方案和任务拆解的方法论参考，除非当前环境明确提供对应 skill 或命令。

5. `GSD / gsd-*` 若当前环境可用，可用于复杂规划、代码审查、执行、验证、UI 评审；不可用时按 `<files>/<action>/<verify>/<done>` 手工推进。

6. 只有实际使用某个 skill / tool，且它影响处理方式或用户需要复现时，才说明用到了什么。

## 原型实现偏好

1. 默认本地运行，优先简单、稳定、易启动。除非用户明确要求，不默认考虑生产部署、监控和复杂权限。

2. 原型阶段可用 mock 数据，但必须与真实接口路径区分，不用 mock 掩盖真实依赖问题。

3. 体验闭环优先：页面能打开，流程能走通，按钮有反馈，状态清楚，空态/错误态合理。

4. 除非用户明确要求，不主动引入数据库、登录系统、支付、部署流水线或复杂状态管理。

5. 代码和文档要便于产品经理理解和后续修改：命名清楚，结构简单，关键假设写明。

## 输出规范

1. 优先表格化  
   只要涉及判断、结论、对比、步骤、风险、文件变更、验证结果，优先用 Markdown 表格呈现，让用户快速扫读。

2. 简单问题短答；复杂问题按“判断 -> 方案 -> 步骤 -> 风险/验证”组织。

3. 用户要方案时，给清晰结构和可复制内容；用户要实现时，说明改了什么、怎么运行、如何验证。

4. 不捏造测试、运行结果或数据来源。没有实际运行就说“未运行”。
